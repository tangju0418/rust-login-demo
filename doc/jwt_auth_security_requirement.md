# authentication_token_and_risk_control_rules

- 状态: draft
- 日期: 2026-03-23

## 1. 范围

1. 本文定义以下业务规则:
   - 登录接口: 用邮箱和密码登录, 返回 jwt token, jwt token 有效时间2小时
   - 刷新jwt接口
   - 获取当前登录主体信息
   - 实现防止暴力破解
   - 用于联调与手动验证的测试 UI 页面
2. 本文仅描述业务规则,不限定语言、框架、存储、部署方式或内部实现。
3. 判断标准:
   - 换语言或技术栈重做时仍必须保留的信息,属于本文范围
   - 仅影响接口封装、内部流程、算法、架构或具体测试执行方式的信息,不属于本文范围

## 2. 基础定义

### 2.1 业务术语

1. `主体标识`: 账号唯一识别字段,本文使用标准化后的邮箱。
2. `access_token`: 用于访问受保护业务能力的访问凭证,有效期固定为 `7200` 秒。
3. `refresh_token`: 用于换取新访问凭证的刷新凭证,不可直接用于访问业务能力,有效期固定为 `2592000` 秒。
4. `风控维度`:
   - `email`: 标准化后的邮箱
   - `ip`: 请求来源地址
   - `email_ip`: 标准化后的邮箱与请求来源地址组合

### 2.2 多源输入选择规则

1. 同名业务字段存在多个来源时,按以下优先级取值:
   - `query` > `body` > `header` > `default`
2. 高优先级来源存在但取值非法时,按非法处理,不得回退到低优先级来源。
3. 多源选择规则适用于登录输入、刷新输入和风控辅助字段读取。

### 2.3 标准化规则

1. `email` 在参与判定前必须完成以下标准化:
   - 去除首尾空白
   - 转为小写
2. 标准化后为空字符串的值视为缺失。
3. 所有时间字段均使用 Unix 时间戳(秒)。

### 2.4 规则优先级

1. 必填字段或字段合法性校验失败时,优先返回参数非法。
2. 风控锁定命中时,优先于账号判定和密码判定。
3. 在同一请求中同时满足多个失败条件时,失败结论优先级如下:
   - 参数非法
   - 风控限制
   - 认证失败
   - 刷新凭证无效

### 2.5 统一失败规则

1. 失败结果统一使用以下结论标识:
   - `invalid_params`
   - `rate_limited`
   - `auth_failed`
   - `refresh_token_invalid`
2. 失败结果统一包含以下字段:
   - `name`
   - `message`
   - `data`
3. 同一失败结论的 `message` 必须保持一致。
4. `invalid_params`、`auth_failed`、`refresh_token_invalid` 的 `data` 使用空对象。
5. `rate_limited` 的 `data` 必须包含剩余限制时长。

## 3. 登录业务规则

### 3.1 触发条件与判定规则

1. 触发条件: 客户端提交登录请求。
2. 登录判定必须同时满足以下条件:
   - `email` 存在且格式合法
   - `password` 存在且非空
   - 请求未命中任一风控锁定
   - 标准化后的 `email` 能定位到账号
   - 账号状态为 `active`
   - `password` 校验通过

### 3.2 输入到输出转换规则

1. 输入字段:
   - 必填: `email`, `password`
   - 可选: `ip`, `user_agent`
2. 输入处理规则:
   - `email` 按标准化规则处理后作为主体标识
   - `ip` 缺失时,使用系统可获取的请求来源地址
   - `user_agent` 缺失时不影响登录判定
3. 输出转换规则:
   - 参数非法时,输出参数非法
   - 命中风控锁定时,输出风控限制,并给出剩余限制时长
   - 账号不存在、密码错误、账号状态不是 `active` 时,统一输出认证失败,不得暴露具体原因
   - 登录成功时,创建新的登录会话并输出新的访问凭证与刷新凭证

### 3.3 边界与例外

1. `email` 的大小写差异和首尾空白差异不得影响账号识别结果。
2. 账号不存在与密码错误的对外结果必须一致。
3. 账号状态不是 `active` 时,不得登录成功。
4. 处于锁定期内时,登录流程不得进入账号判定或密码判定。
5. 未提供 `email` 或 `email` 非法时,本次登录尝试仅按 `ip` 维度参与风控判定。
6. 同一主体允许存在多个并行有效的登录会话。

### 3.4 约束

1. `email`: `string`, 必填,必须满足邮箱格式。
2. `password`: `string`, 必填,不能为空字符串。
3. `ip`: `string`, 可选;缺失时允许由系统补全。
4. `user_agent`: `string`, 可选。

## 4. 刷新业务规则

### 4.1 触发条件与判定规则

1. 触发条件: 客户端提交刷新请求。
2. 刷新判定必须同时满足以下条件:
   - `refresh_token` 存在且非空
   - `refresh_token` 未过期
   - `refresh_token` 未被撤销
   - `refresh_token` 仍归属于原签发主体
   - 主体状态为 `active`

### 4.2 输入到输出转换规则

1. 输入字段:
   - 必填: `refresh_token`
2. 输出转换规则:
   - `refresh_token` 缺失或非法时,输出参数非法
   - `refresh_token` 过期、已撤销、重复使用或主体不匹配时,输出刷新凭证无效
   - 主体状态不是 `active` 时,输出认证失败
   - 刷新成功时,本次使用的旧 `refresh_token` 立即失效,并输出新的访问凭证与新的刷新凭证
   - 刷新成功时,新签发的 `refresh_token` 代替本次使用的旧 `refresh_token` 成为当前有效刷新凭证

### 4.3 边界与例外

1. 已失效的 `refresh_token` 再次使用时必须失败。
2. 同一 `refresh_token` 并发刷新时,最多只允许一次成功。
3. 主体状态从 `active` 变为非 `active` 后,已有 `refresh_token` 不得继续刷新成功。
4. 同一主体再次登录成功后,其他登录会话中的有效 `refresh_token` 继续保持有效。

### 4.4 约束

1. `refresh_token`: `string`, 必填,不能为空字符串。
2. 刷新成功后,触发本次刷新的旧 `refresh_token` 必须立即不可再次用于刷新。
3. `refresh_token` 有效期固定为 `2592000` 秒。
4. 每个有效 `refresh_token` 仅对应其所属登录会话, 不影响同一主体的其他登录会话。

## 5. 获取当前登录主体信息业务规则

### 5.1 触发条件与判定规则

1. 触发条件: 客户端携带 `access_token` 请求当前登录主体信息。
2. 判定必须同时满足以下条件:
   - `access_token` 存在且非空
   - `access_token` 未过期
   - `access_token` 能定位到主体
   - 主体状态为 `active`

### 5.2 输入到输出转换规则

1. 输入字段:
   - 必填: `access_token`
2. 输出转换规则:
   - `access_token` 缺失或非法时,输出参数非法
   - `access_token` 过期、无效、主体不存在或主体状态不是 `active` 时,输出认证失败
   - 判定成功时,输出当前登录主体信息

### 5.3 边界与例外

1. `access_token` 仅可用于读取其所属主体的信息。
2. `refresh_token` 不可用于获取当前登录主体信息。

### 5.4 约束

1. `access_token`: `string`, 必填,不能为空字符串。
2. 成功结果只返回当前登录主体所需的最小必要信息。

## 6. 防暴力破解业务规则

### 6.1 触发条件与判定规则

1. 触发条件: 发生登录尝试。
2. 风控仅作用于登录流程,不作用于刷新流程。
3. 风控判定维度如下:
   - `email`
   - `ip`
   - `email_ip`
4. 统计窗口为 `10` 分钟滑动窗口。
5. 锁定规则如下:
   - `email_ip` 在窗口内连续失败次数大于等于 `5` 时,锁定 `15` 分钟
   - `email` 在窗口内累计失败次数大于等于 `10` 时,锁定 `30` 分钟
   - `ip` 在窗口内累计失败次数大于等于 `30` 时,锁定 `10` 分钟
6. 判定顺序如下:
   - 先判断当前是否已处于锁定期
   - 未锁定时,执行本次登录判定
   - 登录失败后,更新对应维度的失败次数与锁定状态

### 6.2 输入到输出转换规则

1. 输入字段:
   - 登录尝试结果
   - `email`
   - `ip`
2. 输入处理规则:
   - `email` 参与风控前必须完成标准化
   - `email` 缺失或非法时,仅使用 `ip` 维度
   - `ip` 缺失且系统无法补全时,不使用 `ip` 相关维度
3. 输出转换规则:
   - 未命中锁定时,允许进入或继续登录判定
   - 命中锁定时,输出风控限制,并给出剩余限制时长
   - 登录失败时,相关维度失败次数增加
   - `email_ip` 维度在一次登录成功后,其连续失败计数归零
   - `email` 与 `ip` 维度失败计数不因单次登录成功直接清零,仍按统计窗口自然失效

### 6.3 边界与例外

1. 锁定期内的请求必须直接返回风控限制。
2. 锁定到期后,同维度请求恢复正常判定。
3. 风控对外不得暴露命中的具体维度或阈值细节。
4. 同一请求可同时命中多个维度锁定,对外仅返回统一的风控限制结论。

### 6.4 约束

1. 风控限制返回时,剩余限制时长必须为非负整数(秒)。
2. 风控限制的对外失败结论必须统一。

## 7. 数据结构定义

### 7.1 登录输入 `LoginInput`

1. `email`: `string`, 必填
2. `password`: `string`, 必填
3. `ip`: `string`, 可选
4. `user_agent`: `string`, 可选

### 7.2 登录成功结果 `LoginResult`

1. `access_token`: `string`, 必填
2. `access_token_expires_in`: `int64`, 必填,固定为 `7200`
3. `refresh_token`: `string`, 必填
4. `refresh_token_expires_in`: `int64`, 必填,固定为 `2592000`

### 7.3 刷新输入 `RefreshInput`

1. `refresh_token`: `string`, 必填

### 7.4 刷新成功结果 `RefreshResult`

1. `access_token`: `string`, 必填
2. `access_token_expires_in`: `int64`, 必填,固定为 `7200`
3. `refresh_token`: `string`, 必填
4. `refresh_token_expires_in`: `int64`, 必填,固定为 `2592000`

### 7.5 当前登录主体结果 `CurrentUserResult`

1. `id`: `string` 或 `int64`, 必填
2. `email`: `string`, 必填
3. `status`: `enum(active, disabled)`, 必填

### 7.6 失败结果 `ErrorResult`

1. `name`: `string`, 必填
2. `message`: `string`, 必填
3. `data`: `object`, 必填

### 7.7 风控限制附加数据 `RateLimitedData`

1. `retry_after_seconds`: `int64`, 必填

### 7.8 默认失败附加数据 `DefaultErrorData`

1. 空对象, 必填

### 7.9 账号 `User`

1. `id`: `string` 或 `int64`, 必填
2. `email`: `string`, 必填
3. `status`: `enum(active, disabled)`, 必填
4. `created_at`: `int64`, 必填
5. `updated_at`: `int64`, 必填

### 7.10 刷新凭证记录 `RefreshTokenRecord`

1. `id`: `string` 或 `int64`, 必填
2. `user_id`: `string` 或 `int64`, 必填
3. `session_id`: `string` 或 `int64`, 必填
4. `issued_at`: `int64`, 必填
5. `expires_at`: `int64`, 必填
6. `revoked_at`: `int64`, 可选

### 7.11 风控状态 `RiskState`

1. `dimension`: `enum(email, ip, email_ip)`, 必填
2. `dimension_key`: `string`, 必填
3. `fail_count`: `int32`, 必填
4. `window_start_at`: `int64`, 必填
5. `locked_until`: `int64`, 可选

### 7.12 演示测试账号 `DemoUser`

1. `email`: `string`, 必填
2. `status`: `enum(active)`, 必填

## 8. 演示测试账号规则

1. 演示环境必须存在 `2` 个预置测试账号:
   - `test1@brain.im`
   - `test2@brain.im`
2. 两个测试账号状态均为 `active`。
3. 两个测试账号使用统一初始密码为123456。
4. 初始密码不得以明文形式写入代码或文档, 需由运行环境提供。

## 9. 测试 UI 页面业务规则

### 9.1 目标

1. 提供仅用于联调与手动验证的测试 UI 页面。
2. 测试 UI 页面不作为正式业务前台。
3. 测试 UI 页面必须支持对登录、刷新、当前登录主体查询和暴力破解风控规则进行可视化验证。
4. 测试 UI 页面视觉风格应采用清新、简洁的 PC 端页面风格,便于长时间联调和信息查看。

### 9.2 页面能力范围

1. 登录页面必须提供以下能力:
   - 输入 `email`
   - 输入 `password`
   - 触发登录
   - 展示登录成功结果
   - 展示登录失败结果
2. 刷新凭证页面或刷新功能区域必须提供以下能力:
   - 读取当前持有的 `refresh_token`
   - 触发刷新
   - 展示刷新成功结果
   - 展示刷新失败结果
3. 当前登录主体信息区域必须提供以下能力:
   - 读取当前会话中的 `access_token`
   - 触发获取当前登录主体信息
   - 展示当前登录主体信息
4. 暴力破解测试区域必须提供以下能力:
   - 配置用于测试的 `email`
   - 配置用于测试的错误 `password`
   - 展示当前测试将使用的 `ip`
   - 单次触发错误登录
   - 连续触发多次错误登录,用于快速验证锁定阈值
   - 展示每次尝试的结果
   - 在命中风控限制时展示统一失败结论
   - 在命中 `rate_limited` 时展示 `retry_after_seconds`
5. 当前会话信息区域必须提供以下展示能力:
   - 当前 `access_token`
   - 当前 `access_token_expires_in`
   - 当前 `refresh_token`
   - 当前 `refresh_token_expires_in`
   - 最近一次失败结论
6. 页面整体布局必须适配 PC 端浏览器,至少提供以下体验:
   - 清晰区分登录、刷新、当前用户、暴力破解测试和当前会话信息等功能区域
   - 主要操作按钮、输入框和结果展示区域在默认桌面宽度下无需横向滚动
   - 重要结果信息应便于快速扫描,避免不同区域信息混杂

### 9.3 输入到输出转换规则

1. 登录操作成功时,页面中的当前会话信息更新为最新签发结果。
2. 登录操作失败时,页面必须展示失败结论;若页面中已有旧凭证,不得因本次登录失败而覆盖旧凭证。
3. 刷新操作前无可用 `refresh_token` 时,页面必须明确提示当前不可执行刷新。
4. 刷新操作成功时,页面中的当前会话信息更新为最新签发结果。
5. 刷新操作失败时,页面必须展示失败结论,且不得覆盖现有凭证。
6. 获取当前登录主体信息时,页面必须使用当前会话中的 `access_token` 发起请求,不要求手工输入 `access_token`。
7. 获取当前登录主体信息成功时,页面必须展示当前登录主体信息。
8. 获取当前登录主体信息失败时,页面必须展示失败结论,且不得清空现有凭证。
9. 页面刷新后,当前会话信息必须保留。
10. 暴力破解测试区域执行单次或连续错误登录时,页面必须逐次展示每次尝试的返回结果。
11. 暴力破解测试区域在未命中锁定前,每次错误登录应展示统一的认证失败结论。
12. 暴力破解测试区域在命中锁定后,后续尝试必须展示统一的风控限制结论,并展示 `retry_after_seconds`。
13. 暴力破解测试区域的测试操作不得覆盖当前会话中已有的有效凭证。

### 9.4 边界与约束

1. 页面不得展示明文密码回显。
2. 页面不得保留明文密码历史。
3. 页面展示失败结果时,至少必须展示 `name` 与 `message`。
4. 当失败结论为 `rate_limited` 时,页面必须额外展示 `retry_after_seconds`。
5. 页面仅需展示验证业务规则所必需的信息,不要求展示内部实现字段。
6. 暴力破解测试区域不得要求用户手工构造请求报文或手工复制错误结果,应可直接通过页面操作完成验证。
7. 页面风格应保持清新、简洁和信息密度适中,避免使用偏移动端、过度拥挤或强干扰的视觉样式。
8. 页面配色、留白、字体大小和模块层级应有助于在 PC 端快速识别主操作区、结果区和会话信息区。

## 10. 非本期业务范围

1. 第三方登录
2. 多因子认证
3. 验证码或人机挑战
4. 忘记密码与重置密码
